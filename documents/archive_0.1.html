the COMBINE archive version 0.1

This specification describes the "COMBINE archive" format.

A COMBINE archive is a single file containing the various documents (and in the future, references to documents), necessary for the description of a model and all associated data and procedures. This includes for instance, but not limited to, simulation experiment descriptions in SED-ML, all models needed to run the simulations in SBML and their graphical representations in SBGN-ML.

This version of the specification can be found at: http://co.mbine.org/documents/archive_0.1

The latest version of this specification can found at: http://co.mbine.org/documents/archive

Main authors: Nicolas Le Novère, Frank Bergmann, Richard Adams

Acknowledgements: Subscribers of combine-discuss mailing list and attendees of HARMONY2012

History

The initial idea of an archive to exchange model and simulation descriptions has been proposed by Frank Bergmann to support the use of SED-ML, and is described in the appendix D of the SED-ML Level 1 Version 1 specification document.

A preliminary proposal was put together by Richard Adams, Frank Bergmann and Nicolas Le Novère. The proposal was finally discussed by a larger audience at HARMONY 2012, and the global structure agreed upon. Further discussions were held at HARMONY_2013.

Format of the archive

The COMBINE archive is a "zip" file. Zip is a file format used for data compression and archiving. A zip file contains one or more files that have been compressed, to reduce file size, or stored as is. The technical specification of the ZIP format is available from the PKWARE website.

The extension for the COMBINE archive is .omex, for "Open Modeling EXchange format". The format is not yet registered as a mime type, but should be at some point.

Content of the archive

The archive contains:

1) a manifest file, called "manifest.xml", always located at the root of the archive, that describes the location and the type of each data file contained in the archive (including itself).

The location of those files is defined by a relative URI. In the current version of the COMBINE archive, all the files described must be included in the archive itself. It is envisioned that in the future the manifest could list files located elsewhere, using valid and resolvable http URIs.

2) a metadata file, called "metadata.*" (where * means the suitable file extension) containing clerical information about the various files contained in the archive, and the archive itself.

3) all the remaining files necessary to the model and simulation project.

Format of the manifest file

The format of the manifest file called "manifest.xml" is currently very simple. It is formed of two elements, omexManifest and content, the latter carrying two attributes location and format.

omexManifest is the top element of the file, and carry the namespace declaration, which is http://identifiers.org/combine.specifications/omex-manifest (or equivalent versioned version when this will become necessary).

An element content describes one of the files contained in the archive. It carries three attributes.

  • The value of the mandatory attribute location is a relative URI pointing to the content file.
  • The value of the optional attribute format specifies the format of the content file. If the content is in a format which specification is identified by COMBINE, this value is the corresponding Identifiers.org URI. Otherwise, the content format must be identified by a standard URI (resolvable URL or URN) or an Internet media type (aka MIME type).
  • The optional boolean attribute "master" indicates if a file is to be used first to understand and process the content of an archive. Examples are top model description in a composed model, calling the various submodels; simulation description, calling the different model descriptions and data sources used in the experiment. At most one content element per archive may have its master attribute set to "true".

The following example of a manifest file describes the content of an archive, made up of the manifest file itselt, a model encoded in SBML located in a model directory, a simulation experiment encoded in SED-ML, a publication in PDF and a metadata file in the omex metadata format.

<?xml version="1.0" encoding="utf-8"?>
<omexManifest xmlns="http://identifiers.org/combine.specifications/omex-manifest">
    <content location="./manifest.xml" format="http://identifiers.org/combine.specifications/omex-manifest"/>
    <content location="./model/model.xml" format="http://identifiers.org/combine.specifications/sbml"/>
    <content location="./simulation.xml" format="http://identifiers.org/combine.specifications/sedml"/>
    <content location="./article.pdf" format="application/pdf"/>
    <content location="./metadata.rdf" format="http://identifiers.org/combine.specifications/omex-metadata"/>
</omexManifest>

Advised format for the archive metadata

One can include any type of file in a COMBINE archive, and therefore any type of metadata format. However, in the interest of interoperability, and to ease the development of software support for metadata, a recommended format is provided as part of the specification of the archive.

The recommended format is based on several standards developed by other organisations:

  • The Resource Description Format of the W3C, in particular its RDF, Description, Bag and li elements.
  • vCard 4, a file format standard for electronic business cards. (more information to use vCard in RDF)
  • Metadata Terms of the Dublin Core Metadata Initiative, in particular the terms description, creator,created,modified and W3CDTF. more information to use Dublin Core in RDF)

Note that this slightly differs from the controlled annotations of SBML, that use a set of non-standard names describing properties of vCard version 3. The main differences are:

  • "n" becomes "hasName"
  • "Family" becomes "family-name"
  • "Given" becomes "given-name"
  • "EMAIL" becomes "hasEmail", the value of the email becoming a resource
  • "org" and "Orgname" become "organization-name"


The following example is the metadata related to the simulation description contained in the archive.

<rdf:RDF xmlns:rdf="http://www.w3.org/1999/02/22-rdf-syntax-ns#"
                xmlns:dcterms="http://purl.org/dc/terms/"
                xmlns:vCard="http://www.w3.org/2006/vcard/ns#">
    <rdf:Description rdf:about="./simulation.xml">
        <dcterms:description>
This is the description of what the file simulation.xml is about.
        </dcterms:description>
        <dcterms:creator>
            <rdf:Bag>
                <rdf:li rdf:parseType="Resource">
                    <vCard:hasName rdf:parseType="Resource">
                        <vCard:family-name>Bergmann</vCard:family-name>
                        <vCard:given-name>frank</vCard:given-name>
                    </vCard:hasName>
                    <vCard:hasEmail rdf:resource="fbergman@caltech.edu">
                      <vCard:organization-name> California Institute of Technology</vCard:organization-name>
                </rdf:li>
            </rdf:Bag>
        </dcterms:creator>
        <dcterms:created rdf:parseType="Resource">
            <dcterms:W3CDTF>2012-02-01T14:56:11Z</dcterms:W3CDTF>
        </dcterms:created>
        <dcterms:modified rdf:parseType="Resource">
        <dcterms:W3CDTF>2012-02-01T19:20:30+01:00</dcterms:W3CDTF>
        </dcterms:modified>
    </rdf:Description>
</rdf:RDF>
